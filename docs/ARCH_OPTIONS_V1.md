# 方案对比 V1

## 目标
在保证落地速度的前提下，选择可维护、可回滚的自动化实现路线。

## 方案 A：MaaFramework 复用（推荐先做 POC）
- 核心：复用现有自动化框架能力，围绕 Master Duel 任务流做扩展。
- 优点：落地快、基础能力现成、工程结构成熟。
- 风险：框架适配边界受限，游戏 UI 变更时受上游影响。
- 适用：优先验证“能否快速跑通闭环”。

## 方案 B：自研规则引擎（模板匹配 + 状态机）
- 核心：OpenCV 模板匹配 + 明确状态机。
- 优点：可控性高、调试直观、依赖少。
- 风险：初期开发成本更高，需要自建稳定性机制。
- 适用：长期可维护、希望彻底掌控实现细节。

## 方案 C：混合方案（Maa + 自研策略层）
- 核心：底层动作与识别复用框架，上层任务决策自研。
- 优点：兼顾速度与可控性。
- 风险：双层维护复杂度上升。
- 适用：POC 后确认长期演进需求时。

## 对比结论（当前）
- 先做方案 A 的最小 POC（识别任务 + 更新计数 + 领奖决策演示）。
- 若 1-2 个迭代内无法稳定达标，回退到方案 B。
